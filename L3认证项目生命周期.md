# L3 认证项目生命周期（一次项目实际怎么走）

> 本库其余文档讲"各卡组织 L3 要求是什么";本文讲**一次认证项目从立项到拿批准的端到端流程**——谁负责、分几条认证轨、按什么阶段推进、每阶段产出什么、卡在哪、变更后何时要重测。把零散的工具/要求串成一条可执行的时间线。
> 主要依据:U.S. Payments Forum《EMV Testing and Certification White Paper v5.0》(2023-09,见 [`web-docs/SOURCES.md`](./web-docs/SOURCES.md))。该文以美国收单社区为视角,但各卡组织明确声明这些要求**全球适用**;US 特有项(如 2013 强制期、U.S. Common Debit AID)已标注。

## 一、两条认证轨(关键区分)

很多人把"L3"当成单一动作,实际**一个完整上线项目通常包含两条独立的认证轨**:

| 轨 | 测什么 | 频次 | 谁负责 |
|----|--------|------|--------|
| **① 收单主机认证(Acquirer Host Certification)** | 收单主机能否在 **ISO 8583 + Field 55** 里正确承载完整芯片数据(授权 + 清算/结算) | **每个平台一次** | 收单机构 / 处理方 |
| **② L3 终端测试(L3 Terminal Testing)** | 终端连同其与网络间的端到端集成(卡↔终端↔收单↔网络) | 每个"L2 内核 + 支付应用"组合一次 | **收单机构(L3 是收单方责任)** |

- **存在依赖**:轨② 往往要求轨① 先完成。最典型——**Mastercard 的 M-TIP(L3)要求先拿到 NIV(主机认证)批准**。
- 两轨各卡组织都有自己的名称(见下表),没有统一的"EMV L3 证书"。

### 各卡组织两轨命名对照

| 卡组织 | ① 主机认证 | ② L3 终端测试 |
|--------|-----------|--------------|
| Mastercard | **NIV**(Network Interface Validation) | **M-TIP**(Terminal Integration Process) |
| Visa | Visa Acquirer Host Certification | **Visa L3 Testing** |
| Discover | Discover Acquirer Host Certification | **D-PAS 终端 E2E 认证** |
| American Express | Amex Host Certification | **Amex E2E(POS 设备)认证** |

## 二、硬前提(进 L3 之前必须满足)

- 终端已取得**当前有效的 EMVCo L1 + L2 Type Approval**(IFM/内核在有效期内)。
- **过期内核宽限**:各网络普遍支持在内核**过期后两年内**继续用其做 L3 测试;**Mastercard 明确为过期后 12 个月**。内核"重大 vs 轻微变更"的界定见 EMVCo Type Approval Bulletin No. 11。
- **PCI**:POS 设备需 PCI 批准;带 PIN 输入(PED)需 **PCI PTS** 批准。
- 终端**软硬件配置冻结**(固件、支付应用、AID/CAPK/TAC/限额等,见 [终端配置:AID 与 CAPK](./终端配置-AID与CAPK.md))。
- 收单主机/链路就绪并**尽量镜像生产环境**。

> **EMVCo L3 工具框架**:EMVCo 先对第三方 L3 测试工具做合格认定(标准化组件 + 资质流程);各卡组织把**测试用例 + 测试卡镜像**以标准文件格式提供给工具厂商;测试方从厂商处获取工具,再对每个卡组织分别执行 L3。这是"一套工具(如 BTT)可跨品牌驱动,但仍需分别提交"的由来(见 [FIME L3 测试工具深度解析](./FIME-L3测试工具深度解析.md))。

## 三、通用生命周期(以 L3 终端轨为主线)

```
0 前提就绪 → 1 立项/登记 → 2 配置&测试计划 → 3 执行跑测 → 4 结果评审/返工 → 5 提交&签发 → 6 维护/再测
```

权威阶段名(白皮书各卡组织小节通用):**Prerequisites → Initiation → Test Execution → Results → Test Case Validation → Letter of Certification**。

### 阶段 1 — 立项 / 登记
- 选程序并登记;取得项目标识:Mastercard 在 **TSE** 生成**唯一 M-TIP 参考号**(见 [M-TIP TSE 配置详解](./M-TIP-TSE配置详解.md));Visa 接入 **VCMS**(Visa Certification Management Service)或 Visa 确认的主机模拟器。
- **非接前置审批**:Mastercard 非接需先拿 **VPLA**;Amex 非接走 AEIPS/Expresspay 规范。

### 阶段 2 — 配置声明 + 生成测试计划
- 填能力问卷 / 配置描述:**答案必须与 L2 内核能力、与终端实际声明的 EMV 数据对象三方一致**(`9F33`/`9F35`/`9F40` 等,见 [M-TIP TSE 问卷与终端配置参数](./M-TIP-TSE问卷与终端配置参数.md))。
- 工具据此**动态裁剪**测试计划;加载测试 AID/CAPK、TAC、各项限额。

### 阶段 3 — 测试执行
- 用 **EMVCo 合格 + 卡组织确认**的 L3 工具驱动终端跑用例;**BTT** 面向终端品牌测试、**ASTREX** 等面向主机/端到端模拟(模拟发卡行做联机授权)。
- 联机用例真正走 ISO 8583 + Field 55 的一来一回(ARQC→ARPC),见 [ISO 8583 与 Field 55 报文层](./ISO8583与Field55报文层.md) 与 [ARQC/ARPC 联机授权](./ARQC-ARPC联机授权.md)。
- 采集 log / APDU trace / 报文。

### 阶段 4 — 结果评审 + 返工(Results / Test Case Validation)
- 工具按预期结果自动判定;失败用例人工分析(终端决策错?配置错?报文错?)。
- 修复 → 回归;改固件/配置则受影响用例重跑。

### 阶段 5 — 提交 + 签发(产出术语因网络而异)

| 卡组织 | 提交/通道 | 产出 |
|--------|-----------|------|
| Visa | 报告(如 TSEZ)提交 Visa;自我认证路径下 US 收单**可免提交** | 合规记录 |
| Mastercard | **TSE 文件 / M-TIP 参考号** → 服务商 | **LoA**(Letter of Approval) |
| Discover | 经**认证 E2E 服务商**正式验证 | **Letter of Certification** |
| American Express | Amex 评审 + 工具测试(投产前必须完成) | **Letter of Certification** |

### 阶段 6 — 维护 / 再测(见第五节)

## 四、各卡组织的"加速/自助"路径(现代化重点)

各网络都在推流程简化,这是近年最大变化,值得单列:

- **Mastercard**:**M-TIP Fast Track**、**Modular M-TIP**(模块化)、**M-TIP Self-Approval**(自审批)、**U.S. Delegated L3 Testing**(美国委托测试)。
- **Visa**:**CVES**(Chip Vendor Enabled Service,第三方代收单跑 L3)、**Global Acquirer Self-Accreditation Program**(自助认可;US 收单借此**免向 Visa 提交 TSEZ 报告**)。
- **Discover**:**Streamlined D-PAS E2E Test Plan**、**Acquirer Managed Terminal Certification**;非接与接触**同一次认证**完成(含 CDCVM、U.S. Common Debit AID),接触用例在已认证终端加非接时转为回归。
- **American Express**:**Self (Acquirer/Processor) Managed Terminal Certification**,可用 **ATS**(Amex Test System,免费主机模拟器);仅非接可用 **PTE**(Payment Test Emulator,Google Play 下载)。

> 主机认证侧的模拟器/工具:Mastercard **MAS**(双消息)/ **MDFS**(单消息)+ CITR 规范;Amex **ATS**。

## 五、何时需要 L3 再测(Retesting)

白皮书明确的触发条件(满足任一即需与网络重测):

- 引入**新硬件 / 新 EMVCo 批准内核 / 新支付应用软件**,或做了**支付相关配置变更**;
- 终端或基础设施上的**芯片支付应用处理**发生变更;
- 已部署终端的硬/软件**重大修改**或**更换 EMVCo 批准内核**(重大 vs 轻微见 Bulletin No. 11);
- 变更**影响支付应用**(如终端支持的 CVM 改变、新增支付方式);
- **终端→收单→网络的报文**发生影响芯片处理的变更。

白皮书第 4 节按部署形态进一步细分了再测判定(ATM、半集成/独立终端、收单处理平台、VAR、网关、无人值守/加油机 AFD)。

> **不触发新 L3 认证的例外——Faster EMV**(Amex Quick Chip / Discover Quick Chip / M/Chip Fast / Visa Quick Chip,均为"仅联机"方案,省略离线授权):不影响 L2 内核批准;已 L3 认证的 POS 启用它**只需内部回归测试,无需新的 L3 认证**(Amex 在声明"仅加 Quick Chip"后出具 LoA 修订件)。目前仅在美国获批,且 Amex/Discover/MC 不允许在 ATM 上用。

## 六、高频卡点 / 返工点

- **配置与 L2 不一致** = 测试计划错 → 认证无效(最常见命门)。
- **漏了轨① 或前置审批**:M-TIP 没先过 NIV、非接没拿 VPLA / Type Approval 就开跑。
- **"一次测试多处提交"是误区**:工具可统一,但每卡组织程序/参考号/报告通道独立,需分别提交。
- **冻结后又改固件**:触发回归(对照第五节判定)。
- **TAC / 限额配错**:该联机没联机 = 现场风险敞口(见 [TAC/IAC/TVR 决策逻辑](./TAC-IAC-TVR决策逻辑.md))。
- **责任主体**:L3 是**收单方责任**;Mastercard 明确收单方对其(及商户)部署终端的合规/质量/互操作性担责,逾期不修可触发 Category A 违规评估。
