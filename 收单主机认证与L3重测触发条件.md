# 收单主机认证（Host Certification）与 L3 重测触发条件

> L3 认证有**两条线**：①**终端 L3**（终端集成测试，本库多数文档的主题）；②**收单主机认证 / Host Certification**（验证收单方/收单处理商主机与卡组织网络的报文接口，重点是能否把完整芯片数据正确装入 **ISO 8583 Field 55** 并处理授权/清算）。本文梳理后者的各卡组织流程，以及一个高频实务问题——**改了什么才需要重新做 L3？**
> 关联：[ISO 8583 字段 55 与各卡组织 EMV 数据要求](./ISO8583-字段55-跨卡组织要求.md)（主机报文的核心载体）·[各卡组织 L3 认证测试要求一览](./各卡组织L3认证测试要求一览.md)·[Amex 与 Discover L3 认证深度解析](./Amex与Discover-L3认证深度解析.md)·[L3 认证实例：Sunmi T6F10](./L3认证实例-Sunmi-T6F10终端配置剖析.md)
> 来源：U.S. Payments Forum《Current Global Payment Network Requirements for the U.S. Acquiring Community》v5.0（2023-09）。⚠️ 各卡组织自定流程随时变更，精确步骤以卡组织门户最新版为准。

## 一、主机认证 vs 终端 L3：位置与先后

```
卡片 ── 终端 ── 收单方/收单处理商主机 ── 卡组织网络 ── 发卡行
        └ 终端 L3（端到端集成）      └ 收单主机认证（Host Certification）
```

- **测试对象不同**：终端 L3 验证"终端+应用+内核"在真实报文路径下的行为；主机认证验证"收单主机 ⇄ 卡组织网络"的 **ISO 8583 接口**能否正确携带 Field 55 与相关芯片字段（授权 + 清算）。
- **粒度不同**：主机认证**每个平台做一次**（per platform / per endpoint）；新接入的收单处理商端点需补做含芯片数据的主机测试。
- **先后有依赖**：**Visa 明确要求"终端 L3 测试必须在主机测试开始之前完成"**——因为主机测试需要一台**生产就绪终端**来产生真实联机授权报文。
- 美国市场各卡组织主机测试的强制完成时点为 **2013-04**（EMV 迁移期 mandate）。

## 二、各卡组织主机认证流程

### 2.1 Mastercard —— NIV（Network Interface Validation）

验证客户主机与 Mastercard 网络接口，重点：

- **ISO/IEC 8583 接口**；
- **EMV 接触 + Mastercard Contactless（M/Chip）**交易（视客户 profile）；
- 覆盖**授权（authorization）与清算（clearing）**两类处理。

| 要件 | 说明 |
|------|------|
| **测试工具** | 物理芯片卡或卡模拟器 + EMV 卡/终端 trace 功能；须用最新版（清单见 *M/Chip Qualified Test Tools*，Mastercard Connect） |
| **模拟器** | 双消息系统装 **MAS**（Mastercard Authorization Simulator）；单消息系统装 **MDFS**（Mastercard Debit Financial Simulator），需有效 license。在线有人值守可选用 **MTF**（Mastercard Test Facility，但不推荐） |
| **芯片终端** | 需连一台芯片终端/终端模拟器到测试环境收单基础设施，跑芯片交易 |
| **测试规范** | 最新 **CITR**（Customer Interface Testing Reference，Mastercard Connect → Member Publications） |
| **执行** | Mastercard 项目组生成本次 session 的 NIV 用例清单（足以验证接口，但不穷尽所有技术可能场景）；收单方提交模拟器 log，Mastercard 评审、必要时要求修正重跑 |
| **产出** | **TAN（Testing Acknowledgement Notification）**信件——可作为后续实施的"已完成验证"凭据 |

### 2.2 Visa —— Acquirer Host Certification

- 美国收单方须在**所有支持芯片交易的主机平台**上支持完整芯片数据：**Field 55（ICC 相关数据）**及 V.I.P Authorization / V.I.P Full Service 报文中支持 **VSDC**（Visa Smart Debit/Credit）所需的附加字段。
- 首次实现 Field 55 完整芯片数据需**生产环境激活处理参数**（要走 Visa 相应文书）。
- 基础设施/步骤：与 Visa 立项 → 从 Google Play 获取 **VMCP** app 装到 utility card → 取 Visa Online 上的主机测试脚本 → 用**生产就绪芯片设备**（产生联机授权报文）→ 对**每个独立主机平台**做接触 + 非接主机测试 → 通过后获**测试完成信件**。
- **结算测试可选**，仅当支持离线授权时才需要；芯片数据**必须**进授权请求/响应报文，但美国收单方**无须**在清算/结算/退货报文携带芯片数据。
- ⚠️ **先决条件**：L3 终端测试须先完成。

### 2.3 Discover —— Acquirer Host Certification（D-PAS）

主机须满足 *Discover Authorization Interface Technical Specifications* 与 *Discover Sales Data Interface Technical Specifications*。包含两项测试：

| 测试 | 验证内容 |
|------|----------|
| **D-PAS Acquirer Network Online Test** | 按 Discover 授权报文要求正确收发授权请求/响应（含接触与非接芯片数据）；正确处理网络/发卡行返回的**全部**芯片响应数据（含意外数据）；支持的话处理 PIN 管理交易 |
| **D-PAS Acquirer Clearing Test** | 按适用 Discover 清算格式正确生成清算数据文件；按 *Sales Data Interface* 规范正确发送清算文件 |

- **前置**：完成 D-PAS 授权/清算所需主机改造；连接 Discover 指定测试模拟器。
- **发起文档**：*D-PAS Host Certification Request Form*（声明支持的功能与时间线，Discover 据此分配用例）；可向实施经理申请物理测试卡；**CA Security Officer Registration Form**（注册安全官、获取 CA 公钥）。
- **执行**：优先用物理终端 + 测试卡或卡模拟器；无终端时可用 POS 模拟器/交易生成器（若只能产生静态数据，需追加验证密码学场景的用例）。无需提交 log，但需提供用例↔交易一一对应关系，并跑一批 clean test batch + 含芯片交易记录的清算文件。
- **产出**：评审通过后向收单方/处理商/直连商户发**认证信（Letter of Certification）**。

### 2.4 American Express —— Host Certification

- 美国收单方/处理商须按 **AEIPS**（American Express ICC Payment Specification）与 **Expresspay Contactless Specification** 证明可受理芯片卡（接触/非接/移动）。
- 需认证主机连接的**授权与结算**两侧；规范见 `www.americanexpress.com/merchantspecs`。
- 直连 Amex 网络的商户同样需认证。

## 三、各卡组织主机认证速查

| 卡组织 | 流程名 | 关键测试/工具 | 授权 | 清算/结算 | 产出 |
|--------|--------|----------------|------|-----------|------|
| **Mastercard** | NIV | MAS/MDFS 模拟器、CITR、M/Chip QTT | ✅ | ✅ | TAN |
| **Visa** | Acquirer Host Cert | VMCP+utility card、Visa Online 脚本 | ✅（Field 55） | 可选（仅离线授权时结算测试） | 完成信件 |
| **Discover** | Acquirer Host Cert | Network Online Test + Clearing Test | ✅ | ✅ | Letter of Certification |
| **Amex** | Host Cert | AEIPS + Expresspay 规范 | ✅ | ✅ | —（联系代表） |

> 关键共性：主机认证的核心就是 **Field 55 / 等效芯片字段**能否正确收发与解析——这把它和 [ISO 8583 字段 55](./ISO8583-字段55-跨卡组织要求.md) 紧紧绑定。Visa/Discover 都把"先过终端 L3、用生产就绪设备产生真实报文"作为前提。

## 四、何时需要重做 L3（重测触发矩阵）

U.S. Payments Forum 用三档标记判定一次变更是否需重测：

| 标记 | 含义 | 动作 |
|------|------|------|
| ❗ 重大问题 | **需补测/重测** | 与卡组织重新做 required testing |
| 🔍 不需重认证 | 但建议内部按 required testing 自测，发现问题联系卡组织 | 内部测试 |
| ✅ | 仅建议标准内部回归测试 | regression only |

**核心判据：变更是否影响"支付应用的芯片处理逻辑或内核（L2 kernel）"——影响则需重测；不影响则一般免测。**

### 常见变更 → 是否重测

| 变更场景 | 是否重测 | 原因 |
|----------|----------|------|
| 换 **L1 硬件（IFM）**，不影响芯片处理/内核 | ❌ 否（建议内部回归） | minor change |
| 换 **OS 且涉及内核变更**（XP→Win7 带 kernel 改动） | ✅ 是 | 影响内核 |
| 换 **OS 但不涉及内核/芯片处理** | ❌ 否 | 不影响 chip processing |
| **新增 AID** 到现有终端配置 | ✅ 是 | 改变 **L2 配置** |
| 新增 **cash advance / 余额查询**（ATM） | ✅ 是 | 影响授权报文密文数据 → **含主机测试** |
| 新增 **DCC（动态货币转换）/ cash back** | ✅ 是（**Visa 例外：加 DCC 免 L3 重测**） | 影响授权报文 |
| 多通讯方式（Bluetooth/GPRS/拨号）仅通讯变 | ❌ 否 | 通讯类型 out of scope；同 terminal family 一套测试即可 |
| 多终端**厂商**部署 / 改了影响芯片处理的终端配置 | ✅ 是 | per terminal configuration |
| **L2 内核过期** | ❌ 旧机可继续在市（无内核/逻辑变更）；新机须用更新内核并重测 | 见 EMVCo Type Approval Bulletin No.11 |
| 新增 **refund / 语音授权** | ❌ 否 | 视为**非 EMV** 处理 |
| 新增 **gratuity（小费）** | ⚠️ 视情况 | **若改动 CVM** 则需重测 |
| 新增 **PIN Entry Bypass** | ✅ 是 | 影响内核 |
| 升级 **PIN pad 带新内核 / 含读卡器** | ✅ 是 | 含 card reader / 内核变更 |
| 升级 PIN pad 但不动芯片处理（如加 loyalty） | ❌ 否 | 不影响 chip processing |
| 改变**交易路由 / 报文中传输的数据** | ✅ 是 | 端到端测试，影响芯片处理；需卡组织介入 |
| portfolio 变更（ISO 买卖、改 MID/TID） | ⚠️ 视情况 | 若**换网关改变路由**则需测；仅改 TMS 不影响 |

### 半集成 / 独立终端补充

- **半集成**：仅打小票的 POS（不参与支付交易）更换 → 免测；PIN pad ⇄ ECR 间报文无变化则不需与商户重新认证，有芯片处理变更则需。
- **独立终端**：
  - 非支付应用变更、不影响芯片交易的支付应用变更 → 免测（minor）。
  - 收单处理商提供的独立 POS 方案：**首次完整终端测试后，同一收单平台下各商户/各收单方一般无须重测**。
  - 加**带读卡器**的外接 PIN pad → 需重测；加**仅联机** PIN pad → 免测；加**离线** PIN pad → 需重测。
  - **禁用某个 CVM**（原已全 CVM 测过）→ 一般不需重测，建议回归并与厂商确认。
  - LoA 列了多个终端配置 → **同一 terminal family 内一套 required testing 即可**，不必逐配置测。
  - 加 **P2PE** → 一般不影响 EMV 实现，互不影响。

> 这套矩阵正是 L3 工程实务里"该不该再跑一轮 BTT/M-TIP"的判断依据：**凡触及内核、L2 配置（AID）、CVM、授权报文密文数据、交易路由的变更，基本都要重测**；纯硬件/通讯/非支付/路由不变的小改动多可免测。
