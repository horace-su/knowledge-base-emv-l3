# ISO 8583 DE39 应答码全表、软/硬拒绝与拒绝治理（decline recovery）

> DE39（Response Code / Action Code）是"这笔交易放行还是拒绝"的最终裁决位。本文用**卡组织一手来源**梳理：① Visa 官方 Action Code 完整码表；② Mastercard 的拒绝三分组 + Merchant Advice Code（MAC）；③ 软拒绝 vs 硬拒绝的重试红线；④ EMV 3-D Secure / 网络令牌 / Account Updater 这几条拒绝挽回机制的规范出处。
>
> 关联：[实测案例：Visa 非接 59 → 从 55 域 `91` 反解 ARC](../07-实测案例/实测案例-Visa非接59拒绝-ARC反解.md)（RC 59 实战）·[ISO 8583 DE55 逐标签实现清单](./ISO8583-DE55-逐标签实现清单.md)（`91` ARPC+ARC）·[ISO 8583 报文域全景与 POS 录入方式](./ISO8583-报文域全景与POS录入方式.md)（DE39 在报文中的位置）·[收单主机认证与 L3 重测触发条件](./收单主机认证与L3重测触发条件.md)。
>
> ⚠️ 应答码语义**因卡组织而异**，且收单方可能做二次映射；本文码值以下列一手来源为准，落地前仍应对照卡组织门户最新规范。

---

## 一、DE39 定位

DE39 是 ISO 8583 里承载**应答码**的 2 字符字段：`00` = 批准，其余为拒绝或特殊状态。它由发卡行在联机应答中给出，经卡组织网络、收单方回到终端；收单方常把原始码**二次映射/标准化**成自家文案——这正是[实测案例](../07-实测案例/实测案例-Visa非接59拒绝-ARC反解.md)里"59 被显示成『卡片校验错误』"这类误导性提示的由来。

> 交叉印证：发卡行的应答码也会以 ASCII 形态编进 DE55 的 `91`（发卡行认证数据）的 **ARC** 字段——见 [DE55 清单 `91` 速查](./ISO8583-DE55-逐标签实现清单.md)。

## 二、Visa Action Code 完整码表（一手：Visa Developer）

来源：**Visa Developer — Request and Response Codes**（`developer.visa.com/request_response_codes`）。`00`、`10`、`11` 为批准/部分批准，其余为拒绝或特殊条件。

| Code | 含义 | Code | 含义 |
|------|------|------|------|
| `00` | Approved and completed successfully | `62` | Restricted card（本区域/国家不可用）|
| `01` | Refer to card issuer | `63` | **Security violation（来源非正确发卡行）** |
| `02` | Refer to card issuer, special condition | `64` | 交易不满足 AML 要求 |
| `03` | Invalid merchant | `65` | Exceeds withdrawal frequency limit |
| `04` | Pick up card (no fraud) | `70` | PIN data required |
| `05` | Do not honor（笼统拒绝，故意模糊）| `74` | PIN 加密值不一致错误 |
| `06` | Error | `75` | Allowable number of PIN entry tries exceeded |
| `07` | Pick up card, special condition (fraud) | `76` | Unsolicited reversal |
| `10` | Partial approval | `78` | "Blocked, first used"（新卡未正确解锁）|
| `11` | Approved (V.I.P) | `79` | Already reversed (by Switch) |
| `12` | Invalid transaction | `80` | No financial impact |
| `13` | Invalid amount / 币种转换溢出 | `81` | Cryptographic error found in PIN |
| `14` | Invalid account number (no such number) | `82` | **Negative CAM, dCVV, iCVV, or CVV results** |
| `15` | No such issuer | `85` | 无理由拒绝 AVS/CVV2/退货请求（≈批准类）|
| `19` | Re-enter transaction | `86` | Cannot verify PIN（如无 PVV）|
| `21` | No action taken | `89` | Ineligible to receive financial position info (GIV) |
| `25` | Unable to locate record in file | `91` | 发卡行/交换机不可用且无 STIP；超时 |
| `28` | File temporarily not available | `92` | 无法路由（收方机构 ID 无效）|
| `39` | No credit account | `93` | 违反法律，交易不可完成 |
| `41` | Lost card, pick up (fraud) | `94` | Duplicate transaction |
| `43` | Stolen card, pick up (fraud) | `96` | System malfunction |
| `46` | Closed account | `1A` | Additional customer authentication required |
| `51` | Not sufficient funds | `6P` | Verification data failed |
| `52` | No checking account | `B1`/`B2` | Surcharge 金额不允许/不支持 |
| `53` | No savings account | `N0` | Force STIP |
| `54` | Expired card / 有效期缺失 | `N3` | Cash service not available |
| `55` | Incorrect PIN or PIN missing | `N4` | Cash request 超发卡行/批准限额 |
| `57` | Transaction not permitted to cardholder | `N5` | Ineligible for resubmission |
| `58` | Transaction not allowed at terminal | `N7` | Decline for CVV2 failure |
| `59` | **Suspected fraud** | `N8` | 金额超预授权批准额 |
| `61` | Exceeds approval amount limit | `P5`/`P6` | PIN 解锁/修改被拒 |
| — | — | `Q1` | **Card Authentication failed** |
| `R0` | Stop this Payment | `R1` | Stop all future Payments |
| `R2` | 不满足 Visa PIN 资格 | `R3` | Stop all Merchants |
| `Z3` | 无法联机，离线拒绝 | `5C` | Transaction not supported/blocked by issuer |
| `9G` | Blocked by cardholder / contact cardholder | | |

**与"卡片校验失败/校验错误"最相关的几个码**（终端常把它们映射成同一句中文，实则语义不同）：

| Code | 真实语义 | 排查方向 |
|------|----------|----------|
| `59` | 疑似欺诈（发卡行风控决定）| 见[实测案例](../07-实测案例/实测案例-Visa非接59拒绝-ARC反解.md)：拆 `91` 反解 ARC，有 ARPC 即密文链正常 |
| `63` | 安全违规（来源非正确发卡行）| 报文来源/路由/密钥域问题 |
| `82` | CAM/dCVV/iCVV/CVV **卡认证密码学结果为负** | 非接动态卡认证/密文验证真的失败 |
| `Q1` | Card Authentication failed | 卡认证（ODA/密文）失败 |
| `55` | PIN 错误/缺失 | CVM/PIN 流程 |

> 注意：`59`（风控拒绝）与 `82`/`Q1`（真·卡认证失败）是**两类不同**问题。终端一句"卡片校验错误"可能盖住任意一个——**必须回看 DE39 原始码 + DE55 `91`** 才能区分。

## 三、软拒绝 vs 硬拒绝：重试红线

- **软拒绝（Soft Decline）**：临时性，改条件/稍后重试可能成功。典型：`51` 余额不足、`91` 发卡行不可用、`05` Do Not Honor（笼统）。
- **硬拒绝（Hard Decline）**：永久性，**不得用同一卡凭据重试**。典型：`41`/`43` 挂失/被盗、`14` 无效卡号、`R0`/`R1`/`R3` 持卡人撤销授权。

⚠️ **L3 认证测试语境下别套用 CNP 重试逻辑**：测试脚本里的拒绝码若是预期结果，重试只会打乱用例判定（见[实测案例](../07-实测案例/实测案例-Visa非接59拒绝-ARC反解.md) §五）。

## 四、Mastercard：拒绝三分组 + Merchant Advice Code（MAC）

Mastercard 把发卡行原始拒绝**归并为三个伞码**，并附 **Merchant Advice Code（MAC）** 指示商户下一步动作。来源：Mastercard *Network Processing Dual Message Authorization System Guide*（DE39 章节，Mastercard Technical Resource Center）；分组/MAC 明细据处理商公开引用整理。

| 伞码 | 分组 | 归入的原始码（示例）|
|------|------|---------------------|
| **79** | Lifecycle（生命周期）| `14` 无效卡号、`54` 过期卡 |
| **82** | Policy（政策）| `03` 无效商户、`13` 无效金额、`57` 不允许交易、`61` 超取现额、`62` 受限卡、`65` 超取现次数 |
| **83** | Fraud/Security（欺诈/安全）| `41` 挂失、`43` 被盗、`55` 无效 PIN、`63` 安全违规 |

**Merchant Advice Code（DE48 子域）——决定商户能不能重试、怎么救（来源：TabaPay/Qualpay 引 Mastercard 规范）：**

| MAC | 含义 | 建议动作 |
|-----|------|----------|
| **01** | New account information available（需更新信息）| 79/82 组：**查 Account Updater** 拿新卡；83 组：**改用 EMV 3DS 重认证** |
| **02** | Cannot approve now, try later | **约 72 小时后**再试（软拒绝性质）|
| **03** | Do Not try again | **不得重试**；换支付方式（30 天内重试有费）|
| **04** | Token requirements not fulfilled | 换支付方式 |
| **05** | Negotiated value not approved | 换支付来源 |
| **21** | Payment Cancellation | **不得重提**（如订阅已取消）|
| **22** | Merchant not qualified for product code | 核对 MCC/商户类目 |
| **24** | Retry after **1 小时** | 定时重试 |
| **25** | Retry after **24 小时** | 定时重试 |
| **26** | Retry after **2 天** | 定时重试 |
| **27** | Retry after **4 天** | 定时重试 |
| **28** | Retry after **6 天** | 定时重试 |
| **29** | Retry after **8 天** | 定时重试 |
| **30** | Retry after **10 天** | 定时重试 |
| **40** | Consumer non-reloadable prepaid card | 提示客户换卡 |
| **41** | Consumer single-use virtual card number | 提示客户换卡 |

> 💰 **重试收费红线**：卡不在场交易被返回 **MAC = 03 或 21** 后，若在过去 30 天内用**同卡同金额**重试，将被收费（2021-11 生效，2022-01 起 $0.03/次）。**24–30 号 MAC 则明确给出了合规的重试节奏**——这正是"智能重试"该遵循的官方计时，而非任意频次重刷。

## 五、拒绝挽回机制（decline recovery）的一手规范出处

| 机制 | 适用拒绝 | 一手规范 / 说明 |
|------|----------|-----------------|
| **EMV 3-D Secure 2.x**（无感 Frictionless / 有感 Challenge Flow）| 风险类（`59`/`63`/MAC 83）| EMVCo *EMV® 3-D Secure — Protocol and Core Functions Specification*，最新 **v2.3.1.1（2023-05-30）**；规范定义三域模型、frictionless 与 challenge 两条流。**报文字段/`transStatus`/`transStatusReason` 详见 [EMV 3-D Secure AReq/ARes 报文字段与认证结果](./EMV-3DS-AReq-ARes报文字段与认证结果.md)**（其 `transStatusReason 11` 疑似欺诈 / `01` 卡认证失败正对应本文 `59` 与 `82`/`Q1` 之分）|
| **网络令牌（Network Token）** | 生命周期类（换卡/过期）| EMVCo *EMV® Payment Tokenisation Specification – Technical Framework*，公开件含 **Guide to Use Cases v2.2.1（2023-01-16）**、**FAQ v2.3.1（2022-10-24）**；v2.4 为 2026-04 草案（订阅可见）。定义 TSP 角色、令牌生命周期，换卡后台自动更新 |
| **Account Updater（AU）** | 生命周期类 | 卡组织服务（Visa VAU / Mastercard ABU）；对应 **MAC 01** 的"查 Account Updater"动作 |
| **智能路由 / 智能重试** | 系统类 / 笼统拦截（`05`/`91`/`96`）| 收单方/网关能力，非卡组织规范；须受 **MAC 03/21 + 30 天同卡同额收费**红线约束 |

> 卡组织侧的机制（3DS、Network Token、AU、MAC）是**规范定义**的、可引；"Transformer 智能重试"等属收单方产品能力，**无公开码值/阈值**，勿当规范引用。

---

## 六、一句话总览

**DE39 决定生死，但一句中文文案会把语义抹平**——诊断先回原始码：`59` 是风控、`82`/`Q1` 才是真卡认证失败；Mastercard 还要看 **MAC** 决定能不能重试。挽回手段按拒绝类型分派：风险类走 **3DS 加认证**，生命周期类走 **网络令牌 / Account Updater**，系统类走**智能重试（受 30 天重试收费红线约束）**。
