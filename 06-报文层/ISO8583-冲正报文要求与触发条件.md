# ISO 8583 冲正（Reversal）报文要求与触发条件

> 冲正是 L3 主机测试里**最高发的超时/异常场景**，也是收单侧最容易被判失败的一环：交易在联机途中失败或超时后，终端/收单必须发一条 `0400` 把**已扣或可能已扣**的金额撤回，保证持卡人不被重复或错误扣款。本文讲清冲正的 **MTI 与报文形态、触发条件、关键数据元（DE90 原始数据元 / DE95 冲正金额 / 匹配键）、EMV 侧（第二次 GENERATE AC = AAC）与 8583 冲正的衔接、DE55 是否随冲正上送、各卡组织差异，以及 L3 测试怎么造超时验冲正**。
>
> 承接 [ISO 8583 报文域全景与 POS 录入方式](./ISO8583-报文域全景与POS录入方式.md)（MTI/Bitmap/DE 结构与 `0400/0420`）、[DE55 逐标签实现清单 §三（退款/冲正 X400/X420）](./ISO8583-DE55-逐标签实现清单.md)、[DE39 应答码与拒绝治理](./ISO8583-DE39应答码与拒绝治理.md)（`76` Unsolicited reversal / `79` Already reversed）、[TAC/IAC/TVR 决策逻辑](../01-基础概念/TAC-IAC-TVR决策逻辑.md)（两次 GENERATE AC）、[收单主机认证与 L3 重测触发条件](./收单主机认证与L3重测触发条件.md)。
>
> ⚠️ 冲正的**精确 MTI、必填域集、原始数据元子域划分、重发次数与时限**随报文协议与卡组织而异（Visa BASE I/II、Mastercard CIS/IPM、银联 8583、各收单主机手册）。本文给出业界通行框架与主流卡组织取向，精确取值以**你对接的卡组织/收单主机报文规范最新版**为准。

## 一、什么是冲正，与退款/撤销的区别

**冲正（Reversal）= 由受理侧（终端/收单）主动发起、把一笔"未确知结果"或"结果异常"的联机交易撤回**，使账务恢复到交易前状态。它不是一笔新交易，而是对**特定原交易**的否定，因此报文里必须携带足以**唯一定位原交易**的键。

| 概念 | 发起方 | 时机 | 是否需匹配原交易 | 典型 MTI |
|------|--------|------|------------------|----------|
| **冲正 Reversal** | 受理侧（终端/收单）自动 | 原交易**当次/当日**、结果未确知或异常 | **是**（携原始数据元） | `0400`/`0420` |
| **撤销 Void / Cancellation** | 商户操作 | 原交易已成功、尚未清算 | 是（引用原授权） | 视主机，常走 `0400` 或专用 | 
| **退款 Refund / Return** | 商户操作 | 原交易可能已清算 | 否（独立贷记交易） | `0200`（DE3=20） |

> 关键区别：**冲正是"系统级纠错"**（超时/无响应/格式错），由程序自动触发、必须匹配原交易；**退款是"业务级贷记"**，是一笔全新的独立交易。三者在 [DE55 是否上送](./ISO8583-DE55-逐标签实现清单.md) 上也不同——退款多数不带或仅带子集 DE55，冲正见下 §五。

## 二、MTI：冲正请求 / 反转 / 通知

ISO 8583:1987 版（Visa/MC 授权网络仍广泛使用）：

| MTI | 名称 | 语义 | 是否要求应答 |
|-----|------|------|--------------|
| `0400` | **Reversal Request（冲正请求）** | 请求撤回原交易，**等待** `0410` 确认 | 是 |
| `0410` | Reversal Request Response | 交换机/发卡行确认冲正 | — |
| `0420` | **Reversal Advice（冲正通知）** | "先斩后奏"：受理侧已本地撤回，**通知**对端 | 是（`0430`） |
| `0430` | Reversal Advice Response | 确认收到通知 | — |

- **Request（`0400`）vs Advice（`0420`）的实务差别**：`0400` 是"我要撤，等你确认"；`0420` 是"我已撤，告诉你一声"。终端超时无响应时，因为**根本没收到发卡行结论**，通常发 `0400`（请求撤销那条也许发卡行已记账的授权）。
- ISO 8583:**2003** 版对应 `1420`（Reversal Advice）/`1421`（Reversal Advice Repeat）——重发时用 **Repeat** MTI，见 §六。
- 承接 [报文全景篇 §一](./ISO8583-报文域全景与POS录入方式.md)：那里列出 `0400 冲正请求 / 0420 反转` 属同一 MTI 家族。

> **金融冲正 vs 授权冲正**：授权阶段（`0100`）超时冲正撤的是"授权额度占用"；账务/消费阶段（`0200`）超时冲正（"Financial Reversal"）撤的是"记账金额"。两者 MTI 都在 `04xx`，但 DE3 处理码与是否影响清算不同。

## 三、触发条件：什么时候必须发冲正

冲正由**受理侧自动**触发，核心判据是"**扣款可能已发生，但受理侧无法确知或无法完成交易**"。典型触发链：

| 触发场景 | 说明 | L3 是否必测 |
|----------|------|-------------|
| **① 联机响应超时（No Response / Timeout）** | 终端送 `0200`/`0100` 后，在超时窗口内未收到 `0210`/`0110`。发卡行**可能已批准记账**，必须冲正 | ✅ 高发核心用例 |
| **② 收到响应但无法向持卡人完成交易** | 已收到 `00` 批准，但终端此后断电/通信断/打印失败/卡已被拔，交易无法落地 | ✅ |
| **③ 响应报文格式/MAC 校验失败** | 收到 `0210` 但报文不可解析或 MAC 不符，受理侧不敢认，视同未知 | ✅ |
| **④ 部分批准（Partial Approval, DE39=`10`）被拒绝落地** | 发卡行只批部分金额，终端/持卡人不接受，需冲正 | 视程序 |
| **⑤ 卡片在联机后被过早移除 / 无法完成第二次 GENERATE AC** | 见 §四：卡已给 ARQC，终端拿到 ARPC 却无法把 TC 写回卡 | ✅（EMV 专属） |
| **⑥ 重复交易自纠** | 检测到重复发送，撤销多余的一条 | 视主机 |

> **超时是冲正的头号来源**，也是 L3/M-TIP/Visa 主机测试脚本刻意注入的场景（见 §七）。测试通过**故意不回、延迟回、回错**来验证受理侧是否"该冲就冲、冲得对"。

## 四、EMV 侧：第二次 GENERATE AC（AAC）与 8583 冲正的衔接

冲正不仅是外层 8583 的事，EMV 交易流程内部也有对应动作，两者要自洽：

- 联机交易第一次 GENERATE AC 产出 **ARQC**（`9F26`，`9F27`=80）请求联机。见 [TAC/IAC/TVR 决策逻辑](../01-基础概念/TAC-IAC-TVR决策逻辑.md)。
- 若联机**超时/无 ARPC/校验失败**，终端在第二次 GENERATE AC 时应向卡请求 **AAC（拒绝，`9F27`=00）** 而非 TC——交易在卡侧被拒绝、**不产生 TC**，同时外层发 `0400` 冲正掉发卡行侧可能已发生的记账。
- 若终端拿到了 ARPC 但**无法把结果写回卡**（卡已拔/掉电），同样落到 AAC + 外层冲正。
- 对应 [DE55 逐标签实现清单 §三](./ISO8583-DE55-逐标签实现清单.md)："联机拒绝/AAC → `9F26`=AAC、`9F27`=00"。

> 一致性要点：**AAC 路径的 DE55（若随冲正上送）里 `9F26`=AAC、`9F27`=00**，不能还带着 ARQC/TC 的形态；外层 DE39 冲正原因与内层 CID 要讲同一个故事。

## 五、冲正报文关键数据元

冲正报文的灵魂是"**能唯一定位原交易 + 说清撤多少、为什么撤**"。

### 5.1 匹配键（定位原交易）

冲正与原交易必须靠以下域**逐一对应**，否则发卡行无法匹配、返回 `79`（Already reversed）或匹配失败：

| DE | 名称 | 作用 |
|----|------|------|
| **DE11** | STAN（系统跟踪号） | 与原交易同一 STAN（或原 STAN 放进 DE90） |
| **DE37** | Retrieval Reference Number (RRN) | 跨报文检索键，须等于原交易 |
| **DE7/DE12/DE13** | 传输/本地日期时间 | 时间窗内匹配 |
| **DE41/DE42** | TID / MID | 受理点一致 |
| **DE2 / DE4 / DE49** | PAN / 金额 / 币种 | 与原交易一致 |

### 5.2 DE90 原始数据元（Original Data Elements）——冲正的核心域

**DE90** 是冲正/通知类报文的专用域，定长 42 位，把**原交易的关键标识**打包进来，供发卡行精确匹配：

```
DE90 子域（n42）:
  原 MTI(4) + 原 STAN(6) + 原传输日期时间(10) +
  原受理机构 ID(11) + 原转发机构 ID(11)
```

> 有了 DE90，发卡行不必依赖当前报文的 DE11/DE7 即可回指原授权。**DE90 缺失或原 STAN/日期填错，是冲正匹配失败的典型根因。**

### 5.3 DE95 冲正金额 / 替代金额（Replacement Amounts）

**DE95** 承载**部分冲正**的金额信息（定长 42 位，四段各 12 位金额）：

```
DE95 子域（n42）:
  实际交易金额(12) + 实际结算金额(12) +
  实际交易手续费(12) + 实际结算手续费(12)
```

- **全额冲正**：撤回整笔，DE4 等于原金额，通常无需 DE95。
- **部分冲正**：只撤一部分（如原授权 100、实际只需 60，冲正 40，或部分批准场景），用 DE95 告知"实际应保留的金额"。

### 5.4 冲正原因与应答

| DE | 名称 | 冲正场景取值 |
|----|------|--------------|
| **DE39** | Response Code | 请求侧可带**冲正原因码**（如超时/无法完成）；应答侧发卡行回 `00` 接受、`79` 已冲正、`76` Unsolicited reversal（见 [DE39 篇](./ISO8583-DE39应答码与拒绝治理.md)）|
| **DE25** | POS Condition Code | 受理条件（正常/异常） |
| **DE3** | Processing Code | 与原交易一致，标明原交易类型 |
| **DE60/DE90/DE95** | 私有/原始/替代金额 | 见上 |

## 六、DE55 是否随冲正上送？

**没有普适答案**——与退款一样，取决于卡组织与主机接口（见 [DE55 标签分组速查的贯穿性前提](./ISO8583-DE55-标签分组速查.md)）：

- **多数场景**：冲正报文**不携带完整 DE55，或仅带子集**（对齐 [DE55 逐标签实现清单 §三"退款/冲正 X400/X420"](./ISO8583-DE55-逐标签实现清单.md)）。因为冲正撤的是账务，发卡行靠 DE90 匹配原交易即可，无需重新校验完整 EMV 密文。
- **部分卡组织/主机**：要求冲正回带**关键 EMV 标签子集**（如 `9F26` 密文、`9F27` CID、`9F10` IAD、`9F36` ATC、`95` TVR、`9A/9C/9F02`），以证明冲正指向的确是那笔 EMV 交易，或供发卡行做二次密文校验。
- **AAC 场景**：若上送 DE55，形态须为 AAC（`9F26`=AAC、`9F27`=00），见 §四。

> 结论同全库口径：**以你对接卡组织主机接口规范为准**；L3 测试脚本会分别校验"冲正带/不带 DE55"两条路径。

## 七、重发、幂等与"必达"保证

冲正的账务意义要求它**必须最终送达**，因此几乎所有网络都规定重发机制：

- **重发（Repeat）**：`0400` 未收到 `0410` 时按间隔重发，2003 版切到 `1421`（Advice Repeat）MTI，并置 **Repeat 指示**，避免被当重复交易。
- **Store-and-Forward（存储转发）**：链路断时本地落盘冲正，恢复后补送（Mastercard/Discover 等对"必达冲正"有明确要求）。
- **幂等**：发卡行对已处理的冲正返回 `79`（Already reversed）而非再撤一次；受理侧收到 `79` 视同成功。
- **限次/限时**：超过重发上限或当日截止后转人工/差错处理，不能无限重发。

## 八、各卡组织差异（要点）

| 卡组织 | 冲正取向要点 |
|--------|--------------|
| **Visa** | BASE I 授权冲正 `0400`；有专门的 Reversal Reason；`76` Unsolicited reversal / `79` Already reversed（见 [DE39 篇](./ISO8583-DE39应答码与拒绝治理.md)）；超时必须冲正是 L3 主机测试项 |
| **Mastercard** | 明确区分 **Full / Partial Reversal**，部分冲正用替代金额；对"必达冲正 + 重发"有强要求；M-TIP 主机侧会验超时冲正与 DE55 处理 |
| **Discover / D-PAS** | E2E 认证含 Online + Reversal 场景；见 [Amex 与 Discover L3](../03-各卡组织L3认证/Amex与Discover-L3认证深度解析.md) |
| **JCB / Amex** | 授权冲正走各自主机接口的 `04xx`，原始数据元子域以其接口手册为准 |
| **银联** | 8583:2003 体系，冲正/冲正重发有明确报文与 60 域/61 域约定；PBOC 联机流程内 AAC 与外层冲正同样要自洽 |

> ⚠️ 上表为框架取向；精确 MTI、DE90/DE95 子域和重发时限**以各卡组织门户/主机接口规范最新版为准**（本库贯穿口径）。

## 九、L3 测试怎么造超时验冲正

冲正是主机侧集成测试（不是纯终端 L2）的重点，[FIME BTT](../02-fime测试工具/FIME-BTT-品牌测试工具深度解析.md) / M-TIP / Visa 测试集用**模拟发卡行 + 联机响应控制**来注入异常：

| 注入手法 | 借助 | 期望被测系统行为 |
|----------|------|------------------|
| **不回响应（超时）** | 联机响应 XML `emvsim.*` 设"不响应/超时延迟"（见 [EMVCo L3 框架篇](../01-基础概念/EMVCo-L3测试框架与标准化文件格式.md)）| 超时窗后自动发 `0400`，DE90 指向原 `0200`，卡侧走 AAC |
| **延迟到超时边界** | 设延迟 = 超时阈值附近 | 验证超时判定与冲正触发时机 |
| **回不可解析/MAC 错报文** | 构造坏响应 | 受理侧视同未知，发冲正 |
| **回 `10` 部分批准后拒绝** | 设 DE39=`10` | 部分冲正 + DE95 正确 |
| **冲正后再收原响应** | 先超时后迟到 | 幂等：不重复扣、正确处理 `79` |

L3 校验点（对齐 [DE55 多场景脚本](./ISO8583-DE55-逐标签实现清单.md) 与 [跨域一致性](./ISO8583-报文域全景与POS录入方式.md)）：
1. **该冲就冲**：超时/异常后确实产生 `0400`，且在时限内。
2. **冲得对**：DE90（原 MTI/STAN/日期/机构）精确匹配原交易；DE4/DE49 一致；部分冲正 DE95 金额正确。
3. **内外自洽**：卡侧第二次 GENERATE AC = AAC；若带 DE55 则为 AAC 形态。
4. **必达**：未确认时按 Repeat MTI 重发，收到 `79` 正确收敛。
5. **不误冲**：正常批准且完成的交易**不得**触发冲正（假阳性同样判失败）。

## 十、常见失败点速查

| 失败现象 | 根因 | 修复方向 |
|----------|------|----------|
| 超时后**未发**冲正 | 超时判定/冲正触发逻辑缺失 | 补齐 timeout→reversal 状态机 |
| 冲正被发卡行拒绝匹配 | **DE90 原 STAN/日期/MTI 填错**或缺失 | 校正原始数据元打包 |
| 部分冲正金额错 | DE95 四段金额段位错位或全额/部分混淆 | 按接口手册对齐 DE95 子域 |
| 冲正带了 ARQC/TC 形态 DE55 | 未走 AAC，内外不自洽 | 第二次 GENERATE AC 请求 AAC |
| 收到 `79` 仍重复扣/重发 | 幂等处理缺失 | 把 `79` 视同冲正成功、停止重发 |
| 正常交易误发冲正 | 超时阈值过短/响应误判 | 调超时窗、修响应解析 |

## 十一、实现级样本 A：银联系收单主机（Cardnow POS Spec V2.13）`0400/0410` 逐域

> 一手底本：**CardNow POS ISO 8583 主机接口规范 V2.13**（§7.11 Reversal，与 [DE55 逐标签实现清单](./ISO8583-DE55-逐标签实现清单.md) 同源，DCSCC 机密，正文已剥离 PII）。这是一个真实的**银联/亚太收单主机**报文变体，MTI 用 `0400/0410`，值得注意的是它**不用 DE90/DE95**，而是用 **DE60 自定义域的子域**承载原交易标识——印证本库"精确子域以主机接口规范为准"的口径。

**该主机的 TimeoutReversal 逻辑（原文要点）**：终端向主机发出请求后未在预设时间内收到响应，即进入 TimeoutReversal Processing。两种可能——① 请求根本没到主机；② 主机处理了但响应没回到终端。**一种恢复方法覆盖两者**：终端发 `0400`，主机据报文判断此前是否收到过原报文；收到过则负责撤销其财务影响，未收到则无事可做；**两种情况主机都回 `0410` + 应答码 `00`**，终端收到后清除 TimeoutReversal 标志、继续下一笔。

`0400`（请求）/`0410`（响应）逐域（M=必填 / C=条件 / O=可选 / OE=回显）：

| DE | 域名 | 格式/类型 | 请求 | 响应 | 关键约束（原文） |
|----|------|-----------|:----:|:----:|------------------|
| MTI | 消息类型 | N4 BCD | `0400` | `0410` | — |
| **DE3** | 交易处理码 | N6 BCD | M | — | **同原交易** |
| **DE4** | 交易金额 | N12 BCD | M | C | **同原交易** |
| **DE11** | 受卡方系统跟踪号 STAN | N6 BCD | M | M | POS 终端本条冲正流水 |
| DE12/13 | 本地时间/日期 | N6/N4 BCD | M | M | — |
| DE14 | 卡片有效期 YYMM | N4 BCD | C | — | 手工输入卡号必填 |
| DE22 | 服务点输入方式码 | N3 BCD | M | — | — |
| DE23 | 卡片序列号 | N3 BCD | C | — | POS 能取到时存在（=`5F34`）|
| **DE24** | 网络识别码 | N3 BCD | M | M | **同原交易** |
| DE25 | 服务点条件码 | N2 BCD | M | — | `00` |
| DE35/36/45 | Track2 / Track3 / Track1 | LLVAR/LLLVAR | C | — | — |
| **DE37** | 检索参考号 RRN | AN12 ASCII | O | C | **响应侧当 DE39=`00` 时出现** |
| DE38 | 授权码 | AN6 ASCII | — | C | — |
| **DE39** | 应答码 | AN2 ASCII | O | M | 响应侧必填，`00`=冲正成功 |
| DE41/42 | 终端号 TID / 商户号 MID | ASCII | M | M(41)/—(42) | — |
| DE49 | 交易货币代码 | AN3 ASCII | M | — | ISO-4217 |
| DE54 | 附加金额 | LLVAR ASCII | C | — | 小费调整时必填 |
| **DE55** | IC 卡数据域 | LLLVAR | **C** | **C** | 见 [Appendix C / DE55 实现清单](./ISO8583-DE55-逐标签实现清单.md)——**请求与响应均为条件出现** |
| DE56 | 应答信息 | LLVAR ASCII | — | C | DE39=`ZZ` 时作终端报错文案 |
| DE59 | 支付方式 | ANS5 ASCII | M | — | 见附录 G |
| **DE60** | 自定义域 | LLLVAR HEX | M | OE | 含下列子域 |
| **DE60.3** | **原始消息类型** | N4 ASCII | **M** | — | **= 原交易 MTI（本主机的"原始数据元"载体）** |
| **DE60.4** | **原始系统跟踪号** | N6 ASCII | **M** | — | **= 原交易 STAN（匹配键核心）** |
| DE60.1/60.2 | 终端 SN / 批次号 | ASCII | M | OE | 响应回显 |
| DE62/63 | 发票号 / 分期 | LLLVAR | M/C | — | — |

**从这份真实规范提炼的实现要点：**
1. **原交易匹配靠 DE60.3+DE60.4**（原 MTI + 原 STAN），配合 DE3/DE4/DE24"同原交易"——这台主机没有 DE90，**别硬套 DE90**，一切以对接的主机手册子域为准。
2. **DE55 在冲正里是"条件出现"（C）**，请求和响应两侧都可能带——对齐本篇 §六：带则须为 [AAC 形态](./ISO8583-DE55-逐标签实现清单.md)。
3. **响应侧 `0410` 的 DE39=`00` 是冲正闭环信号**；收到才清 TimeoutReversal，否则重发。
4. **DE39=`68`「Transaction timeout. Need to do reversal or query」** 是该主机显式的"该冲正"触发码。
5. **例外——扫码（QR）交易**：应答码 `Q8`（已送发卡行、未得成功响应）**不发冲正**，而是按 **10/30/50 秒发起查询（inquiry）**。冲正逻辑不无差别套用于所有支付方式。
6. **撤销（Void）≠ 冲正**：本主机 Void 走 `0200`/`0120` + 处理码 `020000`（引用原交易），是业务操作；而 Void 自身超时时**再触发 TimeoutReversal（`0400`）**——即"撤销的冲正"，两层要分清。

## 十二、实现级样本 B：Mastercard 双消息授权系统冲正（`0400/0420`）

> Mastercard **Dual Message System（授权侧，Customer Interface Specification / Authorization）**惯例。⚠️ 精确子域/位置以 **Mastercard 授权接口手册（需 Mastercard Connect 登录）** 为准；下表为业界通行结构，供对齐排查。

**MTI 家族**：`0400` Reversal Request / `0410` Response；`0420` Reversal Advice / `0430` Response。重发用 **Advice Repeat**（置 Repeat 指示），对应本篇 §七的"必达 + 幂等"。

Mastercard 把"撤多少、撤哪笔"拆到三个专用域：

| DE | 域名 | 结构 | 作用 |
|----|------|------|------|
| **DE30** | **Amounts, Original** | 原交易金额(12) + 原结算金额(12) | 声明"原交易本来是多少"，部分冲正的对照基准 |
| **DE90** | **Original Data Elements** | n42：原 MTI(4)+原 STAN/DE11(6)+原传输日期时间/DE7(10)+原受理机构 ID/DE32(11)+原转发机构 ID/DE33(11) | 精确回指原授权（与银联样本的 DE60.3/60.4 功能等价，但字段不同）|
| **DE95** | **Replacement Amounts** | an42：实际交易金额(12)+实际结算金额(12)+实际交易手续费(x+n8)+实际结算手续费(x+n8) | **部分冲正**时告知"实际应保留的金额" |

**全额 vs 部分冲正：**
- **全额冲正**：撤回整笔，DE4=原金额、DE30=原金额，通常**不带 DE95**。
- **部分冲正（Partial Reversal）**：只撤一部分（如部分批准 DE39=`10` 后、或授权额>实际结算），**DE30 放原始金额、DE95 放替代（实际保留）金额**，差额即被撤回。

**冲正原因**：Mastercard 在冲正报文里带 **Reversal Reason（常见落在 DE63 网络数据 / DE60 或私有子域，随消息版本而异）**，标明超时/无法完成/持卡人取消等；发卡行侧应答见 [DE39 篇](./ISO8583-DE39应答码与拒绝治理.md)：`79` Already reversed（幂等收敛）、`76` Unsolicited reversal。

**与 M-TIP L3 的关系**：Mastercard 主机侧集成（NIV/TAN，见 [收单主机认证](./收单主机认证与L3重测触发条件.md)）会验超时冲正是否**该冲就冲、DE90 匹配是否精确、部分冲正 DE30/DE95 是否正确、必达重发是否收敛于 `79`**；EMV 侧第二次 GENERATE AC 须为 AAC，与外层冲正自洽（本篇 §四）。

**两样本对照——同一件事、两套字段：**

| 语义 | 银联系（Cardnow V2.13） | Mastercard 双消息 |
|------|--------------------------|-------------------|
| 原交易 MTI | **DE60.3** | DE90 段 1 |
| 原交易 STAN | **DE60.4** | DE90 段 2（=原 DE11）|
| 原交易日期时间 | DE12/13（同原交易） | DE90 段 3（=原 DE7）|
| 原受理/转发机构 | DE24 网络识别码 | DE90 段 4/5（原 DE32/DE33）|
| 原始金额 | DE4（同原交易） | **DE30 Amounts, Original** |
| 部分冲正金额 | DE54 附加/接口约定 | **DE95 Replacement Amounts** |
| 冲正闭环信号 | `0410` DE39=`00` | `0410`/`0430` 接受，或 `79` 已冲正 |

> 一句话：**"匹配原交易 + 说清撤多少"这件事人人都做，但装进哪个域各家不同**——银联样本用 DE60 自定义子域，Mastercard 用 DE30/DE90/DE95。落地务必以**所对接主机接口规范**逐域核对，切勿跨体系照搬。

## 十三、Visa（BASE I / VisaNet 双消息）冲正

> Visa **VisaNet 双消息系统（V.I.P，授权侧 BASE I）**惯例。⚠️ 精确域/子域位置以 **Visa BASE I Technical Specifications / VisaNet 消息接口手册（Visa Online 登录）** 为准。应答码为**一手来源**（Visa Developer，见 [DE39 篇](./ISO8583-DE39应答码与拒绝治理.md)）。

**MTI 家族**：`0400` Reversal Request / `0410` Response；`0420` Reversal Advice / `0430` Response。Visa 区分 **Full Financial Reversal（全额）** 与 **Reversal Advice / Partial Reversal（通知 / 部分）**。

**关键域**（与 Mastercard 同为 ISO 8583 派生，字段号一致但取值/子域以 Visa 手册为准）：

| DE | 域名 | 作用 |
|----|------|------|
| **Field 90** | Original Data Elements | n42：原 MTI(4)+原 STAN(6)+原传输日期时间(10)+原受理机构 ID(11)+原转发机构 ID(11)，精确回指原授权 |
| **Field 95** | Replacement Amounts | 部分冲正的实际保留金额 |
| **Field 39** | Response Code | 一手码表：`76` Unsolicited reversal、`79` Already reversed（Switch 已冲）、`91` 发卡行/交换机不可用或超时——三者都是冲正相关判据（见 [DE39 篇](./ISO8583-DE39应答码与拒绝治理.md)）|
| Field 4 / 25 / 3 | 金额 / POS 条件码 / 处理码 | 与原交易一致 |
| **Reversal Reason** | 冲正原因 | 随消息版本落在 POS/私有域（F60/F62/F63 一类），标明超时/无法完成/持卡人取消——以 Visa 手册为准，勿臆造字段号 |

**Visa 特有要点：**
1. **必达 + 时限**：Visa 要求超时/未完成交易发起冲正，并有**冲正时限与重发（Repeat）**约定；未在窗口内冲正会留下悬挂授权（占用额度）。
2. **部分批准（`10`）→ 部分冲正**：发卡行部分批准而受理侧不落地时，用 Field 95 撤回差额（对齐本篇 §三⑤、§五）。
3. **L3 归属**：Visa **终端 L3（ADVT/CDET / Global L3 Test Set）以 EMV 内核为主，冲正报文正确性在 Visa 收单主机测试（Acquirer Host Testing / VTS）验证**——见 [收单主机认证与 L3 重测触发条件](./收单主机认证与L3重测触发条件.md) 与 [Visa-Global-L3-Test-Set](../04-visa专题/Visa-Global-L3-Test-Set与qVSDC-DM.md)。
4. **EMV 自洽**：超时→第二次 GENERATE AC=AAC，与外层 `0400` 一致（§四）。

## 十四、Amex（GCAG ISO / XML）冲正——附实测能力证据

> 一手证据：**Amex Expresspay L3 Information Report**（`EMVCoL3_01`，同 [Sunmi T6F10 剖析篇](../07-实测案例/L3认证实例-Sunmi-T6F10终端配置剖析.md) 素材）。Amex 授权走 **GCAG（Global Credit Authorization Guide）ISO / XML** 接口。

报告的 **POS Transaction Capabilities** 段直接给出 Amex 接口的冲正/退款能力取向（实测抓取，非推测）：

| 能力 | GCAG ISO & XML | 含义 |
|------|:--------------:|------|
| **Merchant Reversals Support** | ✅ **true** | **商户冲正被支持** |
| Offline Refunds | ❌ not supported | 离线退款不支持 |
| Online Refunds | ❌ not supported | 在线退款不支持 |
| Pre-Authorization Support | false | 预授权不支持 |
| Stand-In Processing | true | Amex 侧 STIP |

**由此得出的关键实务结论：**
1. **在 Amex GCAG ISO 上"退款"不可用，纠错主要靠"冲正"**——这与银联/MC"退款是独立贷记交易"的世界不同（见 §一）。想撤一笔 Amex 交易，走**商户冲正**而非 refund。此点已在 [Sunmi T6F10 剖析篇](../07-实测案例/L3认证实例-Sunmi-T6F10终端配置剖析.md) 标注（"GCAG ISO 报文下离线/在线退款不支持、商户冲正支持"），本篇给出其在冲正体系中的位置。
2. **冲正 MTI/域**遵循 GCAG ISO 8583 的 `04xx` 家族与原始数据元约定，精确子域以 **Amex GCAG 接口规范**为准。
3. **Stand-In（STIP）**：Amex 侧代发卡行应答，超时冲正逻辑要与 STIP 结果协调。

## 十五、L3 范围提示：加"部分批准/冲正"要不要重测？

一个跨卡组织的高频疑问（出处：**U.S. Payments Forum — Payment Network Host and L3 Requirements 2023**）：

> 给 AFD（加油机等）授权**新增部分批准（partial approval）之类服务，是否需要重跑 L3？**
> 答：**不需要**——这类功能属于**非 EMV 交易处理（non-EMV transaction processing）**，不触发正式 L3 重测。

- 换言之：**冲正/部分批准/部分冲正等"报文层账务逻辑"改动，多数落在主机侧集成范畴，而非终端 EMV L3 重测触发项**——与 [收单主机认证与 L3 重测触发条件](./收单主机认证与L3重测触发条件.md) 的"改了什么才需要重做 L3"矩阵一致：动到 kernel/芯片处理才重跑 L3，纯报文/账务改动在主机侧回归验证。
- 但仍须在**收单主机测试**里覆盖冲正正确性（超时注入、DE90/原交易匹配、部分冲正金额、必达重发、`79` 收敛）。
